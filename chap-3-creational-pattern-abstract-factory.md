추상 팩토리 패턴 (Abstract Factory)

- 의도
상세화된 서브클래스를 정의하지 않고도 서로 관련성이 있거나 독립적인 여러 객체의 군을 생성하기 위한 인터페이스를 제공
=> 책에서 이 말이 너무 어려웠다. 언뜻 이해는 되지만 말이 정확한 답을 내지 않은 느낌
구체적인 클래스에 의존하지 않고 서로 연관되거나 의존적인 객체들의 조합을 만드는 인터페이스를 제공하는 패턴
https://gmlwjd9405.github.io/2018/08/08/abstract-factory-pattern.html
=> 여기를 참조해서 다시 정의해보았다, 한마디로 비슷한 애들끼리 공통적인 특성을 통해 모아보겠다고 이해했다.

- 다른 이름
키트(Kit)

- 동기 (예제)
추상 팩토리 클래스인 WidgetFactory와 WidgetFactory를 상속받은 
서브클래스인 MotifWidgetFactory (실제 위젯을 생성하는 연산을 구현) CreateScrollbar 연산은 모티프의 스크롤바를 반환하고,
서브클래스인 PMWidgetFactory (실제 위젯을 생성하는 연산을 구현)에 있는 해당 연산은 프레젠테이션 매니저에 필요한 스크롤바를 생성해서 반환
=> 윈도우 위젯을 누르고 생성할지 수정할지 인터페이스를 누르면 이제 그에 맞는 서브클래스를 통해 반환됨

- 활용성
1. 객체가 생성되거나 구성/표현되는 방식(메서드)과 무관하게 시스템을 독립적으로 만들고자 할 때 => 인터페이스과 구현의 분리
2. 여러 제품군 중 하나를 선택해서 시스템을 설정해야 하고 한번 구성한 제품을 다른 것으로 대체할 수 있을 때 => 제품군끼리의 유사성이 있어 추상팩토리를 상속받아 정의가능하고 그러므로 대체가능
3. 관련된 제품 객체들이 함께 사용되도록 설계되었고, 이 부분에 대한 제약이 외부에도 지켜지도록 하고 싶을 때 => 추상팩토리에서 만들어진 속성이 그대로 유지되어 개발되고, 상속받아 사용하므로
4. 제품에 대한 클래스 라이브러리를 제공하고, 그들의 구현이 아닌 인터페이스를 노출시키고 싶을 때 => 라이브러리들이 구현체가 실제로 이 패턴을 사용할 것 같다, 참고하면 좋을듯

- 참여자
1. AbstractFactory(WidgetFactory): 개념적 제품에 대한 객체를 생성하는 연산으로 인터페이스를 정의합니다.
2. ConcreteFactory(MotifWidgetFactory, PMWidgetFactory): 구체적인 제품에 대한 객체를 생성하는 연산을 구현합니다.
3. AbstractProduct(Window, ScrollBar): 개념적 제품 객체에 대한 인터페이스를 정의합니다.
4. ConcreteProduct(MotifWindow, MotifScrollBar): 구체적으로 팩토리가 생성할 객체를 정의하고, AbstractProduct가 정의하는 인터페이스를 구현합니다.
5. Client: AbstractFactory와 AbstractProduct 클래스에 선언된 인터페이스를 사용합니다.

- 협력 방법
1. 일반적으로 ConcreteFactory 클래스의 인스턴스 한 개가 런타임에 만들어집니다. 이 구체 팩토리(concrete factory)는 어떤 특정 구현을 갖는 제품 객체를 생성합니다.
이 구체 팩토리(concreate factory)는 어떤 특정 구현을 갖는 제품 객체를 생성합니다. 서로 다른 제품 객체를 생성하려면 사용자는 서로 다른 구체 팩토리를 사용해야 합니다.
2. AbstractFactory는 필요한 제품 객체를 생성하는 책임을 ConcreteFactory 서브클래스에 위임합니다.

- 결과
추상 팩토리 패턴을 사용해 얻는 이익과 부담은 다음과 같습니다.
1. 구체적인 클래스를 분리합니다. (추상 팩토리 -> 구체 팩토리 책임 위임)
2. 제품군을 쉽게 대체할수 있도록 합니다. (추상 팩토리 -> 구체 팩토리 
                                          -> 구체 팩토리)
3. 제품 사이의 일관성을 증진시킵니다. (추상 팩토리 상속 및 동일한 제품군을 한 팩토리를 상속해 개발하므로)
4. 새로운 종류의 제품을 제공하기 어렵습니다. (기존 추상 팩토리를 추상하기 어려움, 그 이유는 추상 팩토리가 변경되면 고스란히 상속받은 구체 팩토리들이 영향을 받기 때문)

- 구현
추상 팩토리 패턴을 구현하는 기법은 여러 가지입니다.
1. 팩토리를 단일체로 정의합니다. (갖가지 제품의 종류를 만들어내는 팩토리는 제품군에 대해서 하나면 되는 것 => 단일체 패턴)
2. 제품을 생성합니다. (구체 팩토리에서 각 제품을 위해 팩토리 메서드를 재정의함으로써 각 제품의 인스턴스를 만듭니다. 
제품군이 약간 다르다면 각 제품군을 위한 새로운 구체 팩토리 서브클래스가 필요합니다
많은 제품군이 가능하다면 새로운 구체 팩토리 생성할 필요없이 원형 패턴을 이용해서 구현할 수 있습니다)
3. 확장 가능한 팩토리들을 정의합니다.

- 관련 패턴
1. 추상 팩토리 클래스는 팩토리 메서드 패턴을 이용해 구현되는데, 원형 패턴을 이용할 때도 있습니다.
2. 구체 팩토리 클래스는 단일체 패턴을 이용해 구현하는 경우도 많습니다.
